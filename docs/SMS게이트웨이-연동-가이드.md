# SMS 게이트웨이 연동 가이드 (게이트웨이 앱 개발자용)

> 2026-07-06 작성. 대상: 법인폰 SMS Gateway 앱 개발자.

## 무슨 일이 있었나

- 게이트웨이 앱은 기존에 옹보딩 Supabase의 `messages` 테이블에 **anon 키로 직접 INSERT**하는 방식이었다.
- **2026-06-23**, PII 보호를 위해 DB 전체에 RLS 잠금이 적용되면서(개인정보가 공개 anon 키로 읽히던 문제 수정) anon 키의 쓰기가 전면 차단됐다.
- 그 이후 앱이 수신한 문자가 전송 실패 → **'미전송' 큐에 적체** 중이다 (앱 화면의 "미전송 N건"이 그것).
- 6/22 04:54 이후 서버에 도착한 인입이 0건이다.

## 필요한 변경 (앱 쪽 1곳)

DB 직접 INSERT 대신, 아래 **인증된 API 엔드포인트로 POST** 하도록 전송 대상만 바꿔주세요.

```
POST https://ong-boarding-pi.vercel.app/api/messages/inbound
Content-Type: application/json
x-webhook-secret: <INBOUND_WEBHOOK_SECRET 값 — 관리자(용식)에게 1회성 비밀 채널로 전달받기>
```

**Body (JSON)**

```json
{
  "from": "010-1234-5678",          // 발신 번호 (하이픈 유무 무관)
  "text": "문자 본문",
  "received_at": "2026-07-06T12:34:56+09:00",  // 선택 — 수신 시각(ISO8601). 없으면 서버 시각
  "device_id": "62e7107f9a9b9eea",  // 선택 — 기기 식별자
  "external_id": "앱이 부여한 고유ID" // 선택이지만 강력 권장 — 재전송 멱등성 보장
}
```

**응답**

- `200 { ok: true, message_id: "..." }` — 저장 완료
- `200 { ok: true, dedup: true }` — 이미 저장된 문자 (재전송 무해)
- `401` — 시크릿 불일치 / `400` — from·text 누락 / `500` — 서버 오류 (재시도 대상)

**재시도 규칙**: 2xx가 아니면 큐에 남겨 재시도. `external_id`를 보내면 중복 저장이 절대 발생하지 않으므로 공격적으로 재시도해도 안전. 적체된 '미전송' 큐는 전환 후 "수동 동기화"로 한 번에 흘려보내면 된다.

## 하지 말아야 할 것

- **service_role 키를 앱에 넣는 방식 금지** — 폰 분실·앱 추출 시 DB 전체(개인정보 포함) 유출. anon 쓰기 정책을 다시 여는 것도 같은 이유로 불가(공개 키라 누구나 가짜 인입을 주입 가능).

## 참고

- Heartbeat: 앱의 heartbeat도 기존에 `device_heartbeat` 테이블 직접 쓰기였다면 같은 이유로 차단된 상태. 필요하면 동일 인증의 heartbeat 엔드포인트를 추가해줄 수 있으니 요청 주세요.
- 서버 쪽 처리(지원자 매칭·AI 응대·가용성 추출·수신거부)는 이 API가 저장한 행을 Supabase Database Webhook이 받아 자동 수행한다 — 앱은 저장 성공(2xx)까지만 책임지면 된다.
