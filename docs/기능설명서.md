# 옹보딩(Ongboarding) 기능 설명서

> 코드/도메인 전반의 단일 참조 문서. `lib/README.md`, `lib/agent/README.md`, `app/api/README.md`가 이 문서의 §3(확정 뉘앙스), §9(환경변수)를 참조한다.
> 제품 방향·로드맵은 [`PRODUCT_DIRECTION.md`](../PRODUCT_DIRECTION.md) 참고.

## 1. 제품 개요
- **제품명**: 옹보딩 (Ongboarding)
- **대상**: 시니어(50~70대) 배달원 등 긱워커 채용에 특화된 **B2B SaaS 채용 관리 플랫폼**
- **두 명의 사용자**
  - **지원자(시니어)**: 앱 설치·복잡한 폼 없이 **SMS/채팅(옹봇)** 만으로 채용 절차를 끝낸다.
  - **담당자(매니저)**: 스크리닝·면접 조율·서류 안내 등 반복 업무를 **AI 에이전트가 대행**, 어드민 대시보드에서 모니터링·개입한다.
- **스택**: Next.js 14(App Router) · React 18 · TypeScript(strict) · Tailwind v4 · Radix/shadcn(+일부 MUI/Emotion) · Supabase · Anthropic Claude · SOLAPI/Slack/지오코딩/Google Sheets · 배포 Vercel.

## 3. 핵심 도메인 규칙 — "확정 뉘앙스 절대 금지"  ⭐
**모든 AI 응대·문구·전이 로직의 최상위 규칙.**

- 지원자가 정보를 보내거나 에이전트의 질문에 긍정해도, 그것이 곧 **근무 확정/배정을 의미하지 않는다.**
- 근무 확정은 **매니저가 별도로** 한다. 따라서 에이전트는:
  - "확정됐다 / 배정됐다 / 내일부터 나오시면 된다" 류의 단정을 만들지 않는다.
  - **시작일**은 매니저 확정 후 안내하므로, 에이전트가 먼저 시작일을 묻거나 정하지 않는다(스크리닝 체크리스트에서 시작일 항목이 제거된 이유).
- 이 규칙은 전 stage(exploration/screening/onboarding/active) 공통이며, 프롬프트·시스템 메시지·전이 처리 어디서도 위반하면 안 된다.

## 4. 채용 퍼널 & 에이전트 단계
에이전트 stage(타입: `lib/agent/types.ts`의 `StageName`):

```
exploration → screening → onboarding → active
                                    ↘ paused / abort
```

| 단계 | 의미 | 모델 |
|---|---|---|
| `exploration` | 초기 탐색·관심 확인 | Sonnet 4.6 |
| `screening` | 자격·운영 조건 안내/확인 (체크리스트 7항목) | Sonnet 4.6 |
| `onboarding` | 앱 설치·교육 안내, 배민 아이디 수신 등 (체크리스트 3항목) | Sonnet 4.6 |
| `active` | 온보딩 완료 후 활동 상태 | Sonnet 4.6 |
| `paused` | 매니저가 일시 정지 (에이전트 응답 보류) | — |
| `abort` | 진행 중단 (조건 불충족 등) | — |

- **screening 체크리스트(7)**: 자차 재확인 / 프로모션 종료가능성 안내 / 정산주기 안내 / 공휴일 업무여부 확인 / 본인명의 정산 문제없음 / 업무시간 체계 이해 / 지원자 질문 해소.
- **onboarding 체크리스트(3)**: 앱설치·교육 안내 발송됨 / 배민 아이디 수신 / 만남장소 안내 발송됨. ⚠️ 셋째 항목(만남장소 안내)은 자동 발송/D-day cron이 **미구현**이라 코드상 자동 true로 설정되지 않는다(매니저 수동). onboarding→active 전이는 실제로는 **배민 아이디 수신** 시점에 일어난다.
- 체크리스트는 `job_candidates.agent_state`(JSONB)에 저장. 키 정의·완료 판정은 `lib/agent/checklist.ts`.

## 5. 인입 SMS 처리 흐름
메인 진입점은 **`POST /api/webhooks/supabase-new-message`** (Supabase Database Webhook). 상세는 [`app/api/webhooks/README.md`](../app/api/webhooks/README.md).

```
지원자 SMS → 법인폰 → SMS Gateway 앱
  └─ Supabase REST POST /rest/v1/messages (direction='inbound')
     └─ Supabase Database Webhook → POST /api/webhooks/supabase-new-message
        ├─ 인증: Authorization: Bearer <SUPABASE_WEBHOOK_SECRET>
        ├─ 멱등 가드: classification IS NULL 일 때만 처리
        ├─ phone으로 applicants 매칭
        │   ├─ 매칭됨 → router.runAgentForCandidate (단계별 응대)
        │   └─ 매칭 안 됨:
        │       ├─ isHardSpam → classification='other'
        │       ├─ Haiku triage (conf ≥ 0.7, 배민 지원) → applicants INSERT + apply 폼 SMS + classification='baemin'
        │       └─ 그 외 → classification='pending' (매니저 인박스로)
        └─ triage usage → ai_usage_daily 적재
```

> ⚠️ **멱등 가드 한계**: 위 `classification IS NULL` 가드는 미매칭 경로만 실질 보호한다. **매칭된 지원자 경로는 classification을 채우지 않으므로**, Supabase webhook 재전송(timeout/5xx) 시 같은 지원자에게 에이전트가 재응대·중복 발송할 수 있다. (P0 개선 예정)
>
> (구) `POST /api/messages/inbound`는 SMS Gateway가 Supabase에 직접 INSERT하도록 바뀌면서 현재 미사용. webhook 경로로 대체됨.

## 6. AI 응대 엔진 구조 (`lib/agent/`)
**인입 1건 = 단계 모듈 1회 호출 = Claude API 1회 호출.** 상세는 [`lib/agent/README.md`](../lib/agent/README.md).

```
인입 SMS → router.runAgentForCandidate
  ├─ job_candidates 로드 → 현재 stage 결정
  ├─ stage.process() → Claude 호출 → reply_text + transition + checklist 갱신
  ├─ SOLAPI 발송 → messages INSERT (토큰 비용 포함)
  └─ transitions.applyTransition() → status 갱신 + 자동 발송(GUIDE 등) + Slack
```

| 파일 | 역할 |
|---|---|
| `router.ts` | 진입점. stage 라우팅 + 발송 + transition. 1분 응답 텀(coalesce). |
| `stages/` | 단계별 모듈 (exploration/screening/onboarding/active). Claude `tool_use`. |
| `transitions.ts` | 단계 전이 부수효과 — 자동 발송, status 갱신, Slack. |
| `checklist.ts` / `types.ts` | 체크리스트 키·완료 판정 / 코어 타입. |
| `system-messages.ts` | 자동 발송 멘트 키 조회 + `{{이름}}` 치환. |
| `examples.ts` | DB `prompt_examples`(톤·운영정보·시스템 메시지) → 프롬프트 빌드(60초 캐시). |
| `baemin-triage.ts` | Haiku 4.5 미매칭 SMS 분류 + 하드 스팸 필터. |
| `usage.ts` | Claude usage → `ai_usage_daily` 적재 헬퍼. |

## 8. 데이터 & 마이그레이션
- DB: Supabase(Postgres). 주요 테이블: `applicants`, `messages`, `jobs`, `job_candidates`, `branches`, `site_managers`, `prompt_examples`, `ai_usage_daily` 등.
- 마이그레이션: `docs/migrations/`에 **`YYYY-MM-설명.sql`** 날짜 프리픽스로 **누적 추가**. 적용된 파일은 사후 수정하지 않는다.
- 발송 메시지(`messages`)의 `sms_type`/`cost`는 DB 트리거가 자동 채움.

## 9. 환경변수
코드에서 `process.env.*`로 직접 접근. **실제 값은 `.env.local`(gitignore 대상)에만 두고, 문서/코드/커밋에 값을 적지 않는다.**

| 키 | 용도 |
|---|---|
| `NEXT_PUBLIC_SUPABASE_URL` | Supabase 프로젝트 URL (클라이언트 노출) |
| `NEXT_PUBLIC_SUPABASE_ANON_KEY` | anon key — 브라우저 클라이언트 |
| `SUPABASE_SERVICE_ROLE_KEY` | service role — **서버 전용**, RLS 우회 |
| `SUPABASE_WEBHOOK_SECRET` | inbound webhook 인증(Bearer). 미설정 시 webhook 500 |
| `CLAUDE_API` | Anthropic API 키 |
| `SOLAPI_API_KEY` / `SOLAPI_API_SECRET` / `SOLAPI_PFID` | SOLAPI SMS·알림톡 인증/발신 |
| `SOLAPI_TEMPLATE_APPLY_RECEIVED` 등 6종 | 알림톡 템플릿 ID. 비어있으면 SMS로 자동 폴백. ⚠️ **현재 6종 전부 비어 있어 모든 발송이 SMS**(알림톡 미사용) |
| `SLACK_WEBHOOK_URL` | Slack 알림. 없으면 no-op |
| `NEXT_PUBLIC_KAKAO_CHANNEL_URL` | 완료 페이지 카카오 채널 버튼 |
| `KAKAO_REST_API_KEY` | 카카오 로컬 지오코딩(폴백) |
| `NAVER_NCLOUD_KEY_ID` / `NAVER_NCLOUD_KEY_SECRET` | 네이버 NCloud Maps 지오코딩(주 사용) |
| `GOOGLE_SERVICE_ACCOUNT_EMAIL` / `GOOGLE_PRIVATE_KEY` / `GOOGLE_SHEET_ID` | Google Sheets 연동 |
| `CRON_SECRET` | Cron 인증 시크릿. ⚠️ **현재 코드는 `user-agent: vercel-cron` 검사에 의존하고 CRON_SECRET이 비어 있어 외부에서 위조 가능** — Bearer 헤더로 강제해야 안전(P0 개선 예정) |

## 10. 로컬 개발 & 배포
- 설치: `npm i` → 개발: `npm run dev` → 빌드: `npm run build` → 린트: `npm run lint`.
- 배포: **Vercel**. Cron(`/api/admin/cron/onboarding-reminder`, 매시간)은 `vercel.json`에 등록.
- 자동화 테스트는 없다. 변경 검증은 보통 **`npm run build` + 타입 통과 + 영향 경로 수동 확인**으로 한다. 발송이 일어나는 변경은 실 발송 없는 경로(`agent/test`, `agent/draft`, `playground`, `simulate`)로 먼저 확인.
