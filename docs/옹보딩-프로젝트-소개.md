# 옹보딩 프로젝트 소개 — 왜, 무엇을, 어떻게

> 2026-07-05 · 대상: 옹고잉 전 팀 · 상태: 팀 공유용 초안

## 한 줄 요약

급구가 터지면 **전화 열 통을 돌리는 대신, 문자 한 방을 쏘고 응답이 자동으로 모이면 매니저는 확정 클릭만** 하게 만드는 것 — 그게 옹보딩입니다.

## 1. 왜 시작하는가 — 우리가 몸으로 아는 문제

우리는 몇 년째 같은 문제를 반복해서 겪고 있습니다.

- 오전 6시의 "저 오늘 못 나가요" 문자. 아는 기사 전화 순회 → 카톡방 수소문 → 안 되면 우리가 직접 뜁니다.
- 백업·긴급 대체 수요는 분기에 몇 번이 아니라 **주 단위로** 발생합니다. 지금도 정규직 백업/교육 인력을 월급 주면서 상시 대기시키고 있습니다.
- 지난달 특정 화주사 1곳에서만 음식값 환불·퀵비·교육비·백업비로 **약 200만 원**이 나갔습니다. (일화 기반 수치 — 아래 '비용 원장'으로 정확히 측정 예정)
- 광고로 어렵게 모은 지원자는 확정자 몇 명을 빼면 **전부 버려집니다.** 당근 정책상 미확정 지원자 정보는 재활용도 못 합니다.
- 실제로 우리 DB에는 지원자 640명이 있지만 **76%는 컨택 이력이 0**이고, 383명은 홈페이지로 지원했는데 아무 응대도 받지 못한 채 방치돼 있었습니다.

문제의 본질은 "채용이 안 된다"가 아니라 **"내일 라인에 넣을 수 있는 사람이 누구인지 아무도 모른다"**입니다. 그리고 이 상태의 비용은 용차 프리미엄 → 관리자 현장 투입 → SLA 위약(환불·퀵비) → 광고 재집행 → 번아웃으로 매달 연쇄적으로 새고 있습니다.

## 2. 옹보딩은 무엇인가

**옹보딩 = 배송 인력의 예비전력(預備電力) 시스템.**

전력망이 평시에 예비 용량을 상시 유지하다가 수요가 튀면 즉시 투입하듯이:

- **평시**: 신규 모집 + 기존 지원자 재활용으로 권역/라인별 '투입 가능 벤치'를 채우고, AI가 신선도(누가 이번 주 가능한지)를 유지합니다.
- **긴급·증차**: 버튼 하나로 벤치에 문자 발송 → 가능 응답 자동 수집 → **확정은 매니저 1클릭.**

| 모드 | 시간 예산 | 동작 |
|---|---|---|
| 긴급 투입 (결원 대체+증차) | 시간~하루 | 벤치에 발송 → 응답 수집 → 매니저 확정 |
| 계획 충원 | 일~주 | 내부 풀 우선 매칭 → 부족분만 외부 게시 |
| 평시 유지 | 상시 | AI가 인입 응대·가용성 갱신·휴면 관리 |

두 가지를 분명히 해둡니다.

1. 긴급 대응(SOS)은 가치가 가장 극적으로 증명되는 순간일 뿐, 제품의 본체는 **평시에 벤치를 채우고 썩지 않게 유지하는 것**입니다. 긴급은 평시 관리의 결과로 쉬워집니다.
2. 옹보딩의 AI는 사람을 판단·예측하는 AI가 아니라 **실무를 대신하는 AI**입니다 — 지원자 분류, 자동 응대, 온보딩 안내, 응답 취합, 휴면 처리. 우리가 반복하는 일을 줄이는 게 전부입니다.

## 3. 우리의 접근이 왜 다른가

보통의 순서: 아이디어 → 시장조사 → 개발 → 고객 찾기.
우리의 순서: **실무 → Pain → 목업 → 내부 사용 → (검증되면) 시장.**

우리는 이 문제로 수년간 실제 돈을 잃어온 **첫 번째 고객**입니다. 목표는 "큰 SaaS를 만들자"가 아니라 **"옹고잉의 운영비를 줄이는 내부 운영체계를 만들자"**입니다. 포부는 절반까지 줄이는 것이고, 정확한 목표치는 4주 비용 원장으로 기준선(baseline)을 만든 뒤 숫자로 확정합니다.

내부에서 증명되면 — 같은 고통을 겪는 다른 운수사·배송조직이 "우리도 쓰고 싶다"고 말하기 시작하면 — 그때 외부 확장을 선택합니다.

그리고 새 기능의 유일한 심사 기준은 이것입니다: **"이게 운영비를 줄이는가?"** (예외는 신뢰 복구·컴플라이언스·측정 인프라 세 가지뿐)

## 4. 지금 실제로 동작하는 것 (2026-07-05, 프로덕션 기준)

과장 없이, 오늘 돌아가는 것만 적습니다.

- 홈페이지(Tally) 지원 → **실시간 DB 인입 + AI 첫 응대** — 383명 방치 사태의 원인(배치 인입)을 제거했습니다.
- 인입 문자 **AI 자동 분류·응대** (탐색→스크리닝→온보딩 단계별 진행, "확정 뉘앙스 금지" 규칙 내장)
- 지원자별 **무로그인 맞춤 공고 링크** — 앱 설치·로그인 없이 문자 속 링크로 내게 맞는 공고를 보고 "관심 있음"을 누르면, 그 행동이 자동으로 가용성 데이터가 됩니다.
- 공고에 단가·참고정보를 입력하면 AI가 지원자 질문에 **직접 답변**
- 파이프라인 단계×가용성 필터·정렬·벌크 발송 (이름/맞춤링크 자동 치환)
- 채널별 게시 링크 (알바몬·잡코리아·오픈채팅·지인 소개 — 유입 출처 자동 태깅)

아직 없는 것도 솔직하게: 웨이브 발송, 신뢰점수, 벤치 보드는 **없습니다.** 이번 파일럿 결과가 좋으면 다음 단계로 만듭니다.

## 5. 다음 4주 — 파일럿

1. **방치 지원자 재활성화**: 383명 중 조건 양호한 50~100명을 선별해 맞춤 공고 링크로 재컨택합니다.
2. **급구 건 옹보딩 경유**: 파일럿 기간 중 발생하는 실제 급구·증차 건을 전부 옹보딩으로 처리합니다 — 공고 등록 → 후보 발송 → 확정 클릭.
3. **비용 원장 (팀 협조 요청)**: 긴급 건이 터질 때마다 시트에 4칸만 적어주세요 — 발생일 / 해결 방법(내부 벤치·용차·직접 투입) / 걸린 시간 / 든 비용. 이 원장이 "옹보딩이 실제로 돈을 아꼈는가"를 증명하는 핵심 데이터입니다. (정산 공제 내역은 옹매니징에서 대조해 채웁니다 — 직접 적을 것은 용차 프리미엄과 걸린 시간 정도입니다.)

**성공 기준 (미리 박아둡니다)**

- 재컨택 응답률 8~10% 이상
- 급구 건의 옹보딩 경유율과 건당 처리 시간 (전화 순회 대비)
- 열람 → 관심 → 스크리닝 진입까지 한 바퀴가 실제로 돈다

실무자가 옹보딩에 남겨야 하는 기록은 **딱 두 가지**입니다: ① 확정 시 status 변경(1클릭 — 확정일 기록은 자동), ② 긴급 건 결과를 원장에 기록. 가용성 갱신·유입 태깅·발송 기록은 시스템이 자동으로 합니다.

## 6. 지키는 원칙 (하드 룰)

- **확정 뉘앙스 금지** — 지원자가 "가능해요"라고 답해도 그건 확정이 아닙니다. 확정은 매니저만 합니다.
- **당근에서 지원 링크 유도 금지** — 플랫폼 정책 위반 전례가 있습니다.
- **긴급 수당은 일회성만** — 단가 래칫(한 번 올리면 못 내림)을 방지합니다.
- **좋은 기사를 가용성 확인으로 괴롭히지 않기** — 가용성은 반복 질문이 아니라 행동(링크 열람·관심 클릭·응답)에서 추론합니다.

## 7. 장기 방향 (가설 — 파일럿 후 재논의)

내부 검증 통과 → 인근 권역의 운수사·배송조직 3~5곳과 PoC → 그 결과로 SaaS 또는 운영대행(Managed Service) 중 선택.

배송 인력을 직접 운영하는 조직(운수사·대행사·자체 배송 화주사)은 모두 같은 고통을 겪고 있고, 용차 오픈카톡방에 상주하며 긴급 용차에 리소스를 쓰는 담당자들이 그 증거입니다. 다만 이건 전부 **우리 내부에서 먼저 증명된 다음의 이야기**입니다. 지금 단계의 전부는 5번(다음 4주)입니다.
