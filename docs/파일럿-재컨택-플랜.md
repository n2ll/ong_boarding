# Phase C 재컨택 파일럿 플랜

> 2026-07-06 5차 인터뷰 확정. 발송 전략 = **급구 건 대기**: 인프라를 완비해두고, 실제 급구·증차 건(1~2주 내 예정)이 생기는 순간 실공고를 탑재해 첫 발송한다.

## 1. 대상 선별 (2026-07-06 실DB 기준)

방치 세그먼트: `status='스크리닝 전'` 383명 (전원 전화번호 보유, 전원 무발송).

⚠️ **주의: `created_at`(전원 2026-06)은 Airtable 일괄 임포트 시점이지 지원 시점이 아니다.** 실제 지원 시점은 `airtable_raw`의 '제출일'에 있으며 분포가 넓다: **2024-10 지원자 234명(61%)**, 2025년 80명, 2026년 69명. 선별 정렬과 문구는 이 사실을 전제로 한다.

인입 성격 (2026-07-06 인터뷰): Tally 설문 유입은 대부분 옹고잉을 인지하고 홈페이지 등에서 자발 유입한 사람들(+ 최근 당근 채팅방에서 설문 링크 수령분 일부). 비마트(baemin) 인입과 성격이 다름 — 비마트 인입자는 특정 지점 공고 지원자라 재컨택 문구를 공유할 수 없다(이번 1차 대상 아님).

| 필터 | 인원 |
|---|---|
| 수도권(서울·경기·인천) + 자차 + 지오코딩 완료 | **168명** ← 스탠바이 풀 |
| └ 그중 옹매니징 '현재 활동 중' (2026-07-06 대조: '스크리닝 전' 전체에서 계약 7명+정산 1명=8명) | 발송 제외 |
| 자차 있으나 지역 미상 (주소 보완 필요) | 50명 (2차 후보) |

**1차 발송 = 50명** (서울 우선, **실제 제출일 최신순**). 문구·응답률 검증 후 확대.

```sql
-- 1차 발송 50명 선별 — 정렬은 '실제 지원 시점'(airtable_raw 제출일) 최신순
SELECT id, name, phone, sido, sigungu, vehicle_type,
       COALESCE(airtable_raw->>'제출일', airtable_raw->>'Submitted at') AS applied_at
FROM applicants
WHERE status = '스크리닝 전'
  AND source = 'homepage'
  AND own_vehicle = '있음'
  AND sido IN ('서울특별시', '경기도', '인천광역시')
  AND lat IS NOT NULL
  AND sms_opt_out_at IS NULL
ORDER BY (sido = '서울특별시') DESC,
         COALESCE(airtable_raw->>'제출일', airtable_raw->>'Submitted at') DESC NULLS LAST
LIMIT 50;
```

## 2. 발송 전 체크리스트 (순서대로)

1. **옹매니징 '현재 활동 중' 대조** — `POST /api/admin/ongmanaging/active-check`에 선별 id 목록 전달 → `active` 목록은 발송 제외. 판정 = 두 신호 합집합: ①활성 계약(`contract_status IN ('체결완료','승인대기')`) ②**지난달 확정 정산**(`monthly_settlements` status='confirmed' — "실제로 뛰었다"의 가장 확실한 증거, 계약 상태가 늦게 갱신되는 케이스를 잡음). 2026-07-06 첫 대조: '스크리닝 전' 방치 목록에서 활성 계약 7명 + 정산만 1명 = **8명 발견** — 대조 없이 발송하면 근무 중인 기사에게 재컨택 문자가 나간다.
2. **실공고 등록** — 급구/증차 건을 공고로 등록 (단가·참고정보 포함 — AI가 직접 답변하는 재료)
3. **긴급 건 기록** — 홈 대시보드 '긴급 건 기록'에 발생 등록 (TTF 시작점, 해결 시 비용까지 기록)
4. **문구 최종 승인** (아래 초안)
5. **발송** — 파이프라인에서 대상 선택 → 벌크 발송 (`#{이름}`/`#{맞춤링크}` 치환, 50명/회 제한). 발송 시 `ping_sent` 이벤트 자동 기록됨

## 3. 문구 초안 (승인 필요) — 실제 지원 시점 코호트별 2종

제약: **확정 뉘앙스 금지** · 광고성 표현 회피(본인 지원 이력 기반 안내 프레임 — 전원 `marketing_consent=false`) · 수신거부 안내 포함 · **"지난달" 같은 시점 단정 금지**(대다수가 1~2년 전 지원) · LMS 기준.

**A안 — 기본 (오래된 지원자 포함 전체에 안전. 2024~2025 코호트는 반드시 이것)**

> [옹고잉] #{이름}님, 안녕하세요. 배송 전문기업 옹고잉입니다.
>
> 옹고잉 배송 업무에 관심을 갖고 지원 설문을 남겨주셨던 이력이 있어 연락드립니다. 시간이 좀 지났을 수 있는데요 — 지금 #{이름}님 조건과 맞는 배송 건이 열려 있어 안내드립니다.
>
> 아래 링크에서 조건(단가 포함)을 확인하시고, 해볼 만하다 싶으시면 '관심 있음'만 눌러주세요. 담당 매니저가 확인 후 직접 연락드립니다.
>
> #{맞춤링크}
>
> ※ 상황이 바뀌셨거나 안내가 필요 없으시면 '그만'이라고 답장 주세요. 이후 안내를 보내지 않습니다.

**B안 — 최근 지원자용 (제출일 6개월 이내 코호트만, 짧은 버전)**

> [옹고잉] #{이름}님, 옹고잉입니다. 얼마 전 남겨주신 배송 지원 설문 보고 연락드려요. 지금 조건 맞는 배송 건이 있어 링크로 안내드립니다. 확인 후 '관심 있음'만 눌러주시면 매니저가 연락드립니다. #{맞춤링크} (안내 중단: '그만' 답장)

시스템 동작 (전부 구현됨):
- "관심 있음" 클릭 → availability 자동 갱신 + 매니저 Slack 알림
- 문자 답장("가능해요" 등) → 가용성 자동 갱신 + `ping_reply` 기록(응답 지연 분 단위)
- **"그만" 등 수신 중단 요청 → `sms_opt_out_at` 하드 플래그 + 휴면 전이.** 이후 벌크 발송·디스패치에서 영구 제외(시스템 강제). 단, 지원자가 먼저 보낸 문자에 대한 1:1 응대는 가능 — 완전 삭제·이탈 처리가 아니라 "캠페인 발송 금지" 상태다.
- "당분간 어려워요" 같은 단순 거절 → 휴면(소프트)만 — 추후 선별 재컨택으로 복구 가능

## 4. 급구 건 표준 운영 동선 (파일럿 기간 반복 루프)

**마감시각(closes_at) 규칙 (2026-07-06 확정)**: 긴급 건 마감 = **발송 + 1~2시간** — 그 안에 확정해야 실무자가 다른 업무로 복귀할 수 있고, 마감 도달 시 즉시 플랜B(용차·직접 투입) 발동. 예외: 심야·저녁 발생 건은 지원자 수면 시간을 피해 당일 21~22시 마감 또는 익일 아침 발송+2시간. 마감 임박 미충족 시 '일회성 긴급 수당' 라벨로 1회 재발송(단가 인상 아님 — 래칫 방지). 전제: SMS 게이트웨이 정상(응답 실시간 수집).

```
급구/증차 발생
  → ① 홈 '긴급 건 기록' 등록 (발생 시각 = TTF 시작)
  → ② 공고 등록 (단가 포함) — 긴급 건에 공고 연결
  → ③ 파이프라인 필터 (권역·자차·가용성) → 벌크 발송
  → ④ 응답 수집 (관심 클릭·문자 답장 — 자동 취합)
  → ⑤ 매니저 확정 = status '확정인력' 전환 1클릭 (라인 경험 자동 태깅됨)
  → ⑥ 홈에서 '해결 기록' (해결 방법·든 비용·실소요 분)  = TTF 끝 + 건별 원장
```

용차·직접 투입으로 해결한 건도 **반드시 ⑥까지 기록** — "옹보딩 없이 해결한 비용"이 baseline이 된다.

## 6. 확장 아이디어 (2026-07-06 실무자 제안, 검토됨)

**활동 기사 병행 투입 트랙**: `active-check`의 '현재 활동 중' 목록은 재컨택(구직 안내) 제외용이지만, **긴급 건에서는 오히려 별도 발송 트랙 후보** — 이미 검증·온보딩된 인력이라 벤치 최상위 티어다. 근무시간대가 다르면 병행 가능.
- 1차 필터 (TMS 불필요): 옹매니징 `contracts`에 `working_hours_start/end`·`working_days`가 있어 **계약상 시간대 비겹침** 판단 가능 — 예: 10~11시 중식 건 ↔ 오후·야간 계약 기사에게만 발송.
- 정밀 필터 (실운행 시간): 옹고잉 TMS DB 연동 필요 — Phase D 이후 백로그.
- 주의: 문구는 구직자용 재컨택과 분리("현재 라인과 병행 가능한 백업 건 안내" 톤), 발송 빈도 상한 필수(좋은 기사 괴롭히지 않기 원칙).
