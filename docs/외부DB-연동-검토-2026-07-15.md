# 외부 DB 연동 검토 — 옹매니징 + 옹고잉 TMS (2026-07-15)

옹보딩이 참조할 두 외부 시스템을 종합 검토. **매칭 키 = 전화번호(normalizePhone).** 저장은 파생값·미러만, 개인정보·정산금액·계좌는 절대 미반입.

## 두 시스템의 역할
| 시스템 | 위치 | 성격 | 답해주는 질문 |
|---|---|---|---|
| 옹매니징 | Supabase(연결됨) | 계약·정산 원장 | "지금 계약된 활동 배송원인가? 최근 몇 개월 정산?" |
| 옹고잉 TMS | AWS Postgres `onggoing_prod`(미연결) | 배송 운영 | "실제 스케줄이 배정돼 뛰고 있나? 어느 화주사·거점·물량?" |

## 옹매니징 스키마 (문서 기준, lib/ongmanaging.ts)
- `contracts(worker_id, contract_status)` — 체결완료/승인대기/계약종료
- `delivery_workers(id, phone)` — 전화 매칭원
- `monthly_settlements(worker_id, year, month, status)` — 월정산(금액 컬럼 select 금지)

## 옹고잉 TMS 스키마 (2026-07-15 읽기전용 조사, 16 tables)
- `user`(~546) — 배송원. **phone**·name·birth·role·status·company_id·client_ids·agency_ids ← 매칭원
- `schedule`(~33k) — worker_id·status·**date_to_work**·delivery_segments·alias ← **실제 배차(스케줄 배정) 신호**
- `schedule_client`(~33k) — schedule_id↔client_id
- `delivery`(~174k) — schedule_id·worker_id·pick_up/drop_off_work_place·date_to_work·status·**load_quantity**·location_id ← 실배송건(라인·물량 원천)
- `company`(~34)·`collaborative`(~30, client↔agency)·`company_role` — 화주사·대행사 구조
- `location`(~2087, road·sido·sigugun·lat/lng)·`location_alias`·`mfc`(거점) — 주소·좌표
- 기타: delivery_issue·log_routing·fcm·mobile_device·app_version·alembic_version
- ※ 별도 '배송라인' 테이블 없음 — 라인 = 화주사(client)별 schedule/delivery 반복 패턴으로 파생.

## 무엇을 어떻게 연동할지 (제품 need 기준)
1. **활동중 배송원 여부** (재컨택 제외 + 배지)
   - 신호: 옹매니징 활성계약 OR 최근 정산 [기존] + **TMS 최근 schedule(date_to_work)** — TMS가 '실제 뛰는 중'의 더 직접적 증거(계약상태 지연 보완).
   - 저장: applicants에 파생 boolean/이유/checked_at 캐시(3-상태, false로 뭉개지 않기). 발송 가드는 실시간 이중 방어 유지.
2. **선탑 → 투입 확정 → 실제 배차 퍼널 완성** ★ (기존 '그리고 app' 사각지대 해소)
   - phone→TMS user→schedule 존재 = 실제 배차 시작. 우리 확정(hired_at) → 실제 배차(TMS schedule) → 실배송(delivery) 전 구간 계측.
   - 표시: 상세 패널에 '배차 시작(TMS)' + 선탑 stats에 확정→배차 전환/리드타임.
3. **공고 프리필** (K-3) — TMS company+delivery 집계 = 화주사 라인 현황(상차지·시간대·물량). clients.ongmanaging_shipper_key↔TMS company 매핑 → 등록/초안 시 라인폼(근무시간·상차지·차종) 자동 채움(비운 필드만, 매니저 확정).
4. **정산 요약** — 옹매니징 개월수·최근월만(금액 금지).

## 연결 방식
- 옹매니징: 기존 Supabase REST 어댑터(lib/ongmanaging.ts) 확장.
- **TMS: 신규 lib/tms.ts (pg 드라이버) + 주기적 sync cron**(하루 1~2회, KST 아침). Vercel→AWS RDS 매요청 연결 지양. phone 매칭으로 applicants에 파생 신호 캐시 + clients에 라인 요약. 미구성 시 안전 스킵.
- env: TMS_DB_* (.env.local 기입·활성). Vercel에도 투입 필요. **크레덧 커밋·노출 금지.**

## 단계
- Phase 1: 활동중 신호(옹매니징+TMS) → 재컨택 정확도·배지
- Phase 2: 배차 계측(TMS schedule) → 선탑→투입→배차 퍼널
- Phase 3: 공고 프리필(TMS company/delivery → 라인폼 자동채움)

## 병행 투입 원칙 (2026-07-15 확인)
한 사람이 시간대가 다르면 여러 배송라인(화주사 무관)에 병행 투입될 수 있다(실운영 케이스). 따라서:
- 확정은 '한 사람=한 투입'이 아니다 — 확정해도 다른 열린 공고의 진행 후보를 지우지 않는다(병행 라인일 수 있음). 죽은(마감) 공고 링크만 위생 정리.
- AI 중지는 사람 단위가 아니라 '확정한 그 공고' 후보만 paused (다른 라인 스크리닝은 계속).
- TMS schedule도 한 worker가 여러 개 가질 수 있음 — 배차 계측은 count/최근일 기준.

## 리스크
- 전화 매칭 품질(번호변경·법인폰) → 캐시는 표시용, 발송 가드는 실시간·checked_at 신선도 노출
- TMS 스키마 드리프트 → env 테이블명 오버라이드, cron 실패는 에러 로그(조용한 실패 금지)
- PII/금액 → 저장은 boolean·개월수·요약뿐. delivery 174k 원본 미복사, 화주사별 파생 집계만.
