# 옹보딩 (Ongboarding) - 제품 방향성 및 기획 문서

## 1. 제품 개요
- **제품명**: 옹보딩 (Ongboarding)
- **목적**: 시니어(50~70대) 배달원 등 긱워커(Gig-worker) 채용에 특화된 B2B SaaS 채용 관리 플랫폼
- **포지셔닝(2레이어)**: **시니어 특화 = 진입 wedge·브랜드·운영노하우**, **공급 커버리지·데이터 = 긱워커 전반(50대 미만 포함) 수용.** 시니어로 깊게 진입하되 인력풀 분류·매칭은 나이가 아니라 능력·가용성 기준으로 설계한다. (배경: §5.3)
- **핵심 가치**: 
  - **시니어 친화적 UX**: 지원자(시니어)가 복잡한 앱 설치나 폼 작성 없이, 친숙한 '채팅(옹봇)'만으로 채용 절차를 마칠 수 있게 돕습니다.
  - **담당자 업무 자동화**: 채용 담당자(매니저)의 단순 반복 업무(스크리닝, 면접 조율, 서류 안내)를 AI 에이전트가 대신 처리하여 리드타임을 획기적으로 단축합니다.

## 2. 주요 기능 및 현재 구현 상태
현재 프론트엔드 UI 골격과 핵심 라우팅이 모두 연결되어 있습니다.

> ⚠️ **구현 상태 주의**: 아래 목록은 **UI·라우팅 연결** 기준이다. 백엔드 실동작 중 일부는 **부분/데모** 상태다 — 예: 알림톡(현재 100% SMS 폴백), 소싱 광고지표(데모), 자동화 탭의 워크플로우 캔버스(목업). 자동화 **점검 규칙 엔진**은 2026-07 Phase A에서 cron(매시 30분) 등록되어 실구동. '매니저 확정'은 실시간 응대의 확정 대기 큐 + 파이프라인 status 변경으로 이원화되어 있으며(전용 화면 없음), 확정 슬롯 기록은 상세 패널 모달 경유 시에만 남는다.

1. **대시보드 & 리포트 (`/`, `/reports`)**: 채용 퍼널 전환율, 소요 기간, 월별 트렌드를 차트(Recharts)로 시각화
2. **AI 에이전트 & 실시간 응대 (`/live`, `/brain`)**: 
   - **에이전트 두뇌**: AI의 페르소나, 예외 처리 규칙(Fallback), RAG(사내 지식베이스 문서 업로드) 설정 뷰
   - **실시간 응대**: AI와 지원자 간의 대화를 실시간 모니터링하고, 필요 시 담당자가 '개입(Human Takeover)'할 수 있는 하이브리드 채팅 콘솔
3. **인재풀 및 파이프라인 (`/pipeline`, `/sourcing`, `/recommendations`)**:
   - **Drag & Drop 칸반 보드**: 지원 접수/대기 ➔ AI 스크리닝 중 ➔ 스크리닝 완료 ➔ 확정 인력 단계로 직관적인 이동 (이 제품엔 면접/캘린더 단계가 없다 — SMS 스크리닝 후 매니저가 확정)
   - **AI 인재 추천 & 소싱**: 제휴 플랫폼의 인재풀을 검색하고 특정 공고에 맞는 지원자를 매칭 (매칭률 제공)
4. **채용 운영 관리 (`/jobs`, `/branches`, `/clients`, `/team`, `/settings`)**: 
   - 채용 공고, 화주사(고객사), 지점, 사내 권한 및 환경설정 관리 리스트

## 5. 인력풀 → 공급망 전환 로드맵 (전략 방향)

> 이 섹션은 "보유한 배송원 인력풀을 어떻게 자산화할 것인가"에 대한 제품 방향이다. §1~4(채용 ATS)는 **인력풀을 채우는** 깔때기이고, 이 섹션은 그 풀을 **운영 가능한 공급망으로 전환**하는 다음 단계다.

### 5.1 한 줄 방향
지금 옹보딩은 **채용 깔때기(ATS)** 다. 다음 단계는 이를 **가용성 기반 공급망(Active Driver Pool)** 으로 한 칸 옮기는 것이다.
핵심 전환은 단 하나 — 인력풀의 상태를 **'채용 진행 단계(status)'와 '공급 가용성(availability)' 두 축으로 분리**하는 것.

### 5.2 스코프 경계 ⭐ (오해 금지)
- **옹보딩이 책임지는 곳: '가용성(availability)'까지.** 누가 이번 주/오늘/지금 일할 수 있는지를 **정확히 알고 최신으로 유지**하는 시스템.
- **옹보딩 밖(외부 TMS·수기 등): 실제 운행·배차·정산·CS.** 출근율·지각률·가동률·운행이력 같은 **운영 지표는 "운행 이벤트"가 있어야 산출**되는데, 옹보딩에는 그 이벤트가 없다(채용 확정 후 추적 종료).
- 따라서 옹보딩 단독으로 만들 수 있는 건 **"가용성 프록시"**(ping 응답·희망슬롯·최근 컨택)이고, 진짜 가동률 지표는 **외부 운행 데이터 연동이 선행 전제**다. → 5.6 결정 필요 사항 참조.

### 5.3 보정된 전략 결론 (메모 검토 반영)
1. **해자는 "500명"이 아니라 "빈 자리를 빠르게 메우는 운영 신뢰성"이다.** 긱워커 공급 자체는 부족하지 않다. 가치는 머릿수가 아니라 *조건별 즉시 투입 가능성*과 *대체 투입 속도*에서 나온다.
2. **단, "즉시 대체"와 "높은 가동률"은 트레이드오프다.** 즉시 대체엔 잉여(놀고 있는 기사)가 필요하고, 잉여는 가동률의 적이다. 해법은 (a) 멀티호밍 전제(응답률·이동시간이 SLA의 진짜 변수) + (b) 권역 밀도. 이 비용 구조를 인정하고 설계한다.
3. **니치는 옹고잉이 이미 경험 있는 반복형 B2B 배송으로 좁힌다.** 타깃 기사는 **시니어에 한정하지 않고 긱워커 전반**(50대 미만 포함)을 포괄한다. 니치 선택 기준은 *기사의 운전 실력*이 아니라 *시간 약속·응대·재배정·대체*가 핵심인 영역 → **"기업 대상 주간 정기 소형 배송"**(예: 정기 도시락·의료/샘플 정기편) 1~2개 버티컬부터. 7개로 분산하면 각 버티컬에서 밀도가 죽어 "즉시 투입 2명" 문제가 재현된다.
   - 참고: 시니어 풀은 *주간 가용성·정시성*에서 이 니치와 궁합이 좋은 **한 세그먼트**이지, 타깃의 전부가 아니다. 젊은 긱워커는 야간·고강도·피크 변동 커버에서 보완적이다.
4. **포지셔닝은 2레이어로 분리한다 (특화 ↔ 수용 모순 해소).**
   - **공급 측(누구를 모으나) = 긱워커 전반.** 화주는 "시니어로 채워줘"가 아니라 "이 조건(야간·피크·고강도 포함)을 채워줘"를 원한다. 시니어 한정은 커버리지 구멍을 만든다. → 데이터모델·매칭은 **나이가 아니라 능력·가용성**(권역·차종·시간대·즉시투입) 기준. 나이 하드코딩 분기 금지.
   - **수요·브랜드 측(어떻게 알려지나) = 시니어 특화 wedge.** 시니어를 잘 다루는 운영 노하우(저마찰 SMS·정시성·응대)가 해자. 초기 GTM은 **좁게**(시니어 wedge + 1~2 버티컬) 간다.
   - 주의: "전반 수용"은 *역량을 열어둔다*는 뜻이지 *지금 다 모은다*가 아니다. 초기 수익은 TAM이 아니라 **권역·시간대 밀도**에서 나오므로(2 참조), 일찍 넓혀 밀도를 흩뜨리지 않는다.
5. **가용성=자동, 확정=매니저.** SLA(빠른 대체)와 최상위 도메인 규칙("확정 뉘앙스 금지 / 확정은 매니저")을 양립시키려면, 가용성 수집은 AI·자동(ping·링크)으로, **배정 확정만 매니저 1클릭**으로. 매니저 확정 루프 속도 = SLA 상한임을 인지한다.
6. **증명 1순위 = 내부 구인비 절감.** 외부 영업·AX 매출은 시간이 걸린다. 반면 내부 구인비 절감(당근·알바몬·헤드헌팅 → 내부 충원율로 대체)은 *지금, 내부에서, 측정 가능하게* 실현되는 가장 확실한 ROI. **영업·AX 운영대행은 북극성**으로 두되 현재 단계 목표로 끌어오지 않는다.

### 5.4 목표 매핑
| 목표 | 방향 |
|---|---|
| 1. 내부 인력풀 잘 활용 | `availability` 상태축 신설 → "지금 쓸 수 있는 사람"을 항상 안다 |
| 2. 인력풀이 구인 정보를 쉽게 파악 | **pull 채널 신설** — 현재는 SMS push 일방향뿐, 인력풀이 활성 공고를 스스로 볼 창이 없음 |
| 3. 내·외부 풀로 효율 구인 + 외부 의존도↓ | `recruit_mode=internal` 우선 소진 → 부족분만 외부. KPI=내부 충원율 |
| 4. 메모(수익화) | 위 1~3으로 "가용성 풀"을 만들고, **운행 데이터 연동 후** 가동률·영업 매트릭스·AX로 확장 |

### 5.5 단계별 로드맵 (각 단계 = 검증 가능한 목표)
```
[1] 영업/현황 자산화 (가장 싸고 빠름 — 데이터 이미 있음)
    지역×차종 인력풀 매트릭스 화면 (sigungu/sido × vehicle_type 집계)
    검증: "서울 평일오전 자차 보유 N명" 같은 실수치가 0이 아닌 값으로 집계됨

[2] 가용성 상태축 (Active Driver Pool 서사의 코어)
    applicants에 availability(가동/이번주가능/즉시가능/휴면) 도입 + 6지표 대시보드
    검증: '즉시 가능' 카드가 실데이터 기반 실수치로 뜨고, 시간 경과로 휴면 전이됨

[3] pull 채널 — 목표 2의 결손 (저마찰: 앱·로그인 없음)
    개인 무로그인 토큰 링크 "내 맞춤 공고" 페이지 + SMS 안내
    검증: 링크 열람 → "관심 있음" 클릭 → availability 갱신까지 한 바퀴 (확정 뉘앙스 금지 유지)

[4] 내부 우선 매칭 — 목표 3
    recruit_mode=internal 흐름 + scoring에 가용성·응답성 가중치 추가
    검증: 한 공고에 내부 추천 → 컨택 → 내부 충원율 KPI 집계

[5] (전제 충족 시) 운영 데이터 연동 — 목표 4
    외부 운행 데이터 연동 → 진짜 가동률/출근율/대체투입 지표 → 화주 영업·AX
    검증: 외부 연동 후 "월 1회 이상 운행 N명(MAU)" 산출. ※ 5.6 결정 선행
```

### 5.6 지표 — 지금 가능 vs 외부 연동 필요 (메모의 6지표·깔때기 분해)
| 지표 | 지금 가능? | 비고 |
|---|---|---|
| 등록 기사 수 / 지역별 / 차종별 | ✅ 즉시 | `applicants` 집계만 |
| **내부 충원율** (증명 1순위 KPI) | ✅ 즉시 | `jobs`×`job_candidates`의 source/내부추천 집계. 외부 의존도↓ 직접 지표 |
| 최근 활동 / 이번주 가능 / 즉시 투입 | ⚠️ 가용성 프록시 | SMS ping 응답·링크 클릭으로 수집 |
| 가동률 / 출근율 / 지각률 / CS / 운행이력 | ❌ 외부 필요 | **운행 이벤트** 없으면 산출 불가 |

> 메모의 KPI("월 1회 이상 운행", "MAU 150~250", "주3회 70명")는 전부 마지막 줄(운행 데이터)에 의존한다. 가용성 프록시는 그 1차 대용일 뿐, 진짜 가동률로 승급하려면 외부 연동이 필요하다.

### 5.7 결정 필요 사항 (Open)
- **운행/배차 데이터의 출처와 연동 시점·주체.** 옹보딩은 가용성까지만 책임하기로 했으므로, 가동률 지표 실현은 외부(TMS/수기) 데이터를 *언제, 누가, 어떤 키로* 옹보딩 인력풀과 연결할지에 달려 있다. → **§6.2에서 단기 보류로 확정.**
- **첫 타깃 버티컬 1~2개 최종 선택** (5.3-3의 "주간 정기형" 후보 중).
- **가용성 신선도(decay) 정책.** 가용성은 시간이 지나면 썩는다. ping 주기·무응답 시 휴면 전이 기준·재확인 주기를 정해야 '즉시 투입 가능' 수치가 거짓이 되지 않는다.
- **ping/broadcast 비용 관리.** §5.5 [2][3]의 가용성 수집은 SOLAPI 건당 과금이 직접 발생한다. 누구에게 얼마나 자주 보낼지(타깃팅·빈도 상한)를 정해 "풀 유지비 > 절감액"이 되지 않게 한다.

## 6. 실무자 검증 및 재정의 — "예비전력 시스템" (2026-07-03)

> 8개 영역 병렬 코드 감사 + 실DB 분석 + 실무자 인터뷰 3라운드의 결론. §5의 전략 방향을 실무 고충("급한 땜빵/증차를 최소 리소스로 즉시 확정") 중심으로 검증·구체화한다. **§5.5의 [2] 가용성 축과 [3] pull 채널은 로드맵 후반이 아니라 첫 실전 투입 그 자체로 승격되었다.**

### 6.1 실사용 실태 (2026-07-03 기준 실DB·인터뷰)
- **옹보딩은 실무에서 사용된 적이 없다.** 실제 채용 스택: 당근(유료 구인) → 당근채팅+전화(스크리닝) → 옹고잉 앱(온보딩·스케줄) → 옹매니징(계약·정산). DB의 6월 데이터(640명)는 파일럿 흔적.
- 인력풀 640명 중 489명(76%)은 컨택 이력 0. '스크리닝 전' 적체 383명은 전원 homepage(Tally→Airtable 배치) 인입 무응대 방치분. 7월 발송 2건.
- 실지원자(job_candidates 151명)는 전원 UI에서 숨겨지는 `__system__` 더미 공고 귀속 — UI 노출 실공고 2건은 후보 0명.

### 6.2 확정된 제약 (인터뷰)
| 제약 | 결정 |
|---|---|
| 당근 문자로 지원 링크 유도 = 당근 내부정책 위반 | 당근 인입 자동화 **신규 투자 중단** (웹훅·시스템 공고는 레거시 유지만) |
| 옹고잉(앱 설치·운행 이벤트) 연동 | 같은 회사 제품이나 **단기 보류**. 옹보딩 스코프 끝 = 확정 기록 + 옹고잉 앱 가이드 발송 (§5.2 재확인) |
| 충원 수요 = 화주사 계약 따라 폭증하는 스파이크형 | 풀의 가치 = "터졌을 때 광고 없이 빨리 채우는 속도" |
| 지점 개념 없는 공고가 다수 (배민 비마트만 지점 단위) | 지점은 공고의 **optional 속성**. 실무 단위는 **라인/권역**. 슬롯보드는 배민형 하위 뷰로 강등 |
| 투입 자격은 라인마다 다름 | 벤치는 **라인 단위** — applicants에 라인 경험 이력 태깅 |
| 긴급 추가 보상은 **일회성만** 가능 (단가 래칫 방지) | 비금전 인센티브(콜 1순위·정기 전환 우선) 병행 |

### 6.3 재정의: 옹보딩 = 배송 인력의 예비전력 시스템
> 평시엔 **삼중 인입**(신규 모집·재활용·긴급 투입 전환)으로 라인/권역별 벤치를 채우고 에이전트가 신선도를 유지하며, 결원·증차가 터지면 버튼 하나로 검증된 벤치에 웨이브 발송 → 가능 응답 수집 → **매니저 1클릭 확정**까지 끝낸다.

같은 인력풀 위의 3가지 운영 모드 (시간 예산만 다름):

| 모드 | 시간 예산 | 동작 |
|---|---|---|
| **긴급 투입** (결원 대체 + 증차 통합) | 시간~하루 | `shift_request` → 벤치 웨이브 발송 → 선착 응답 → 1클릭 확정 + 나머지 자동 마감 안내 |
| **계획 충원** | 일~주 | 공고 등록(단가·정산 포함) → 내부 우선 매칭 + 부족분만 외부 게시 (`recruit_mode`) |
| **평시 유지** | 상시 | 에이전트가 ping·availability 갱신·휴면 전이·벤치 승급 관리 |

- **North star: TTF(요청→확정 리드타임) · 충족률.** 내부 충원율·풀 신선도는 하위 지표로 정렬.
- 진짜 경쟁자는 채용 플랫폼이 아니라 **용차 오픈카톡방**(무검증 스팟 마켓, 다수 기업 경쟁). 차별화 = 검증된 이력·타깃 발송·응답 이력 축적. 오카방은 SOS 게시문구+고유 지원 링크의 **배포 채널로 편입**.
- 외부 모집·긴급 투입의 부산물(미확정자·용차 기사)이 내부 풀의 원천 — **용차→풀 전환 훅**(투입 종료 후 등록 링크 문자)으로 긴급 투입 1건 = 벤치 +1명.

### 6.4 핵심 설계 요소
1. **온보딩의 재정의 = 벤치 사전 승급(프리보딩).** 당일 땜빵에 신규 지원자는 못 쓴다. 스크리닝 통과자에게 확정 전 교육·앱 설치를 미리 안내해 승급. Tier 0(온보딩 완료+최근 가용 확인=콜 가능) / Tier 1(스크리닝 완료) / Tier 2(리드). 문구는 "기회가 오면 바로 시작할 수 있게 미리 준비" 톤 — **확정 뉘앙스 금지 규칙 준수**.
2. **공고 ≠ 요청.** 모집 공고(수명 몇 주)와 배정 요청(수명 몇 시간)은 다른 엔티티 — `shift_request`(라인·일시·차종·인원·자격요건·일회성 수당·마감시각) 신설. jobs 재활용 금지.
3. **웨이브 발송.** 전원 동보가 아니라 신뢰점수순 1차 5명 → 무응답 시 다음 웨이브. 응답은 "가능 의사 수집"일 뿐, 확정은 매니저.
4. **신뢰 점수 = 응답률·응답속도·이행 이력(매니저 기록).** 운행 데이터 연동 없이 옹보딩 안에서 산출 가능 — §5.6 "외부 필요" 지표의 실용적 우회.
